ASEAN Summit 2019

เคสสตัดดี้การออกแบบระบบ registration สำหรับงานประชุมระดับนานาชาติ ที่ต้องคุมความถูกต้องของข้อมูล delegate ข้ามประเทศ ข้ามหน่วยงาน และลดความเสี่ยงของข้อมูลสำคัญที่แก้ผิดไม่ได้ง่าย

Year 2018
Contributor Three, P'Ann, Look Kleaw, P'WIN
Context Ministry of Thailand
Category Diplomatic workflow, registration system
Problem

ระบบเดิมเร็วเกินไปสำหรับข้อมูลที่ผิดพลาดไม่ได้

โจทย์ของ ASEAN Summit 2019 คือระบบลงทะเบียน delegate ที่หลายประเทศส่งข้อมูลเข้าหา host กลาง ถ้าข้อมูลชื่อ ตำแหน่ง บทบาท หรือข้อมูลเดินทางผิด ความผิดพลาดจะกระทบหลายหน่วยงานทันที

ปัญหาใหญ่คือข้อมูลสามารถถูกส่งเข้ากลางได้โดยยังไม่มีการตรวจระดับประเทศ ทำให้ host กลายเป็น single point of failure และต้องรับภาระแก้ไขหลังบ้านในบริบทที่มีความอ่อนไหวสูง

Single point of failure

ข้อมูลจากหลายประเทศไหลเข้ากลางโดยยังไม่มี layer ตรวจสอบก่อนถึง host

Sensitive data

ชื่อ ตำแหน่ง และรายละเอียดเดินทางผิดได้ยาก เพราะมีผลต่อพิธีการและหน่วยงานปลายทาง

Manual habits

ผู้ใช้จำนวนมากคุ้นกับ email-based workflow ทำให้ระบบใหม่ต้องสะท้อนวิธีประสานงานจริง

Multi-agency pressure

ระบบต้องรองรับ 25+ government agencies ที่มี protocol และจังหวะทำงานต่างกัน

Solution

เพิ่ม DAO validation layer เพื่อกระจายความรับผิดชอบก่อนข้อมูลถึง host

ระบบใหม่กำหนดบทบาทหลักเป็น Host, DAO และ Delegate โดย DAO หรือ Delegate Affairs Officer ทำหน้าที่ตรวจและยืนยันข้อมูล delegate ของประเทศตัวเองก่อนส่งถึง host กลาง

ความเสียดทานที่เพิ่มขึ้นไม่ใช่ความซับซ้อนส่วนเกิน แต่เป็น safeguard ที่ช่วยลด upstream errors ทำให้การ track status, issue handling และ error recovery ชัดขึ้นในสภาพแวดล้อมหลายประเทศ

Challenges & solutions

ความยากของงานนี้อยู่ที่ต้องเริ่มสร้างระบบก่อนทุกอย่างนิ่ง

Pending brand identity

CI และ logo ยังไม่ final แต่ development ต้องเริ่ม จึงใช้ centralized stylesheet / pre-design system เพื่อเปลี่ยน branding ทีหลังได้ทั้งระบบ

Low digital literacy

ทีม host คุ้นกับงาน manual และ email จึงต้องทำ contextual inquiry เพื่อ map flow ประสานงานจริง ไม่ใช่ flow ตามสมมติฐาน

Speed vs accuracy

stakeholder อยากส่งข้อมูลเร็ว แต่ข้อมูล diplomatic ผิดไม่ได้ จึงต้อง defend data holding area และ DAO approval ให้เป็น friction ที่จำเป็น

Outcome

ผลลัพธ์คือ registration flow ที่คุมความถูกต้องโดยไม่ฝากภาระไว้ที่ host คนเดียว

  • เปลี่ยน flow จาก Host รับข้อมูลโดยตรง เป็น Host → DAO → Delegate → DAO Recheck → Host Review
  • ทำให้แต่ละประเทศมีจุดตรวจข้อมูลก่อนเข้าสู่ระบบกลาง ลดภาระแก้ไขหลัง submission
  • เพิ่ม visibility ของ invitation status, delegate verification และ need-attention requests
  • ช่วยให้ระบบรองรับหลายภาษา หลายบทบาท และหลายระดับ trust ในบริบทประชุมระหว่างประเทศ
  • ยืนยันหลักคิดว่า UX ที่ดีไม่ได้ลบ friction เสมอไป แต่ต้องวาง friction ให้ถูกจังหวะเพื่อปกป้องระบบ
Process evidence

จาก email-based workflow สู่ flow ที่ตรวจสอบย้อนกลับได้

ในระบบข้อมูลอ่อนไหว friction ไม่ได้แปลว่า UX แย่เสมอไป บางครั้ง friction คือ กลไกปกป้องความถูกต้อง
